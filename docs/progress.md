## 阶段 1B 完成记录

- 已新增 `backend/` Flask App Factory 基础工程，采用 `app/common` 与 `app/blueprints` 目录，符合 `docs/architecture.md`；健康 Blueprint 注册于 `/api/v1/health`。
- 已初始化 Flask-SQLAlchemy、Flask-Migrate、Flask-CORS、环境配置、请求 ID、统一成功/错误响应、API 异常处理、基础日志与 pytest fixture。开发数据库为受控本地 SQLite，测试数据库为独立内存 SQLite；生产配置要求由环境变量提供 MySQL URL、密钥和 CORS 来源。
- 已执行：Python 3.12 虚拟环境创建、生产/开发依赖安装、`flask --app run.py db init`、`flask --app run.py db upgrade`、`pytest`、`flask --app run.py run`、健康检查与 API 404 的实际 `curl` 请求。Alembic 迁移系统已初始化；因尚无业务模型，未生成无意义 revision。
- 验证结果：当前 pytest 6 项通过；健康检查返回 `status: ok`、`service: yingmo-backend`、当前环境和真实 `database: connected`；不存在 API 返回统一 JSON 404。数据库探测失败路径亦由测试覆盖，返回 503 且不会伪称已连接。
- 未完成：前端健康检查调用与连接状态展示（1C）、静态首页迁移、认证及全部业务域；本阶段未建立用户/媒体或任何业务模型。

## 阶段 1C 完成记录

- 采用 `VITE_API_BASE_URL` + Flask CORS 的直接请求方案；未配置 Vite 代理，符合既定架构。开发示例使用 `http://127.0.0.1:5000/api/v1`，后端允许 `localhost:5173` 和 `127.0.0.1:5173`，生产来源仍由环境变量决定。
- 已新增 `frontend/src/api/health.js`、`hooks/useHealthStatus.js` 和 `components/common/BackendStatus.jsx`。首页加载一次真实健康检查，成功时展示服务名称、环境和数据库；失败时区分数据库异常、网络不可达、超时、404、500 和响应异常，并提供手动“重新检查”。
- 后端健康响应明确禁止缓存；CORS 测试覆盖两个本地开发来源。后端 pytest 共 6 项通过；前端 `npm run lint` 与 `npm run build` 通过。
- 当前前端未配置测试框架，且架构将前端自动化测试安排在后续质量阶段；因此本任务未引入额外测试依赖。组件和 API 模块按可独立 mock 的边界拆分。真实浏览器已验证正常连接和后端关闭后的错误展示；本地 in-app 浏览器在服务恢复后的跨域重试存在不一致现象，尽管直接浏览器访问健康 URL、curl CORS 请求与后端测试均成功，需在目标浏览器环境复核恢复按钮。
- 未完成：静态首页迁移、认证及所有业务功能。
