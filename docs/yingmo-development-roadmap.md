# 映墨 Yingmo 开发计划

> 依据：当前 `InkSaku/Ying-Mo` 静态前端原型 + 《映墨 PRD V2》  
> 前端：React + Vite  
> 后端：Flask  
> 执行原则：一个 Codex 任务对应一个小型 Pull Request；每项任务必须可运行、可测试、可验收。

---

## 1. 当前项目基线

### 1.1 已经完成

当前仓库已经具备：

- 映墨 Logo、favicon 和品牌素材；
- 静态 HTML 首页；
- 响应式 CSS；
- 深浅主题和本地主题记忆；
- 首页导航、Hero、上传入口、最近更新、照片墙、评论预览、关于和页脚；
- 7 条写死在 JavaScript 中的照片数据；
- 照片卡片动态渲染；
- 照片详情弹窗；
- 点击遮罩、关闭按钮和 Esc 关闭；
- Vite 开发、构建和预览命令；
- 旧版生活照片分享 PRD。

### 1.2 尚未完成

- React 工程；
- Flask 后端；
- 数据库；
- 用户注册和登录；
- 真实图片上传；
- 日常章节；
- 真实日常发布；
- 游戏、英雄、地图和教材；
- 点赞、评论、收藏；
- 搜索；
- 个人中心；
- 通知；
- 管理后台；
- 自动化测试；
- 生产部署。

### 1.3 现有代码处理原则

- 不直接删除当前静态版本；
- 开发开始前建立 `static-v1` 标签或备份分支；
- 现有页面作为 React 迁移的视觉和交互基线；
- 现有 `galleryItems` 只用于迁移阶段的临时展示；
- 后续真实功能接入后必须删除或隔离硬编码业务数据；
- 新版 PRD V2 是开发依据，旧 PRD 只作为历史资料。

## 3. 总体里程碑

| 里程碑 | 结果 |
|---|---|
| M0 | 文档、架构和仓库规则冻结 |
| M1 | React 与 Flask 均可启动并连通 |
| M2 | 用户可以真实注册和登录 |
| M3 | 用户可以上传并持久保存图片 |
| M4 | 日常生活区形成完整可用闭环 |
| M5 | 点赞、评论、收藏和个人中心可用 |
| M6 | 游戏、英雄和地图知识结构可用 |
| M7 | 游戏教材发布、浏览和筛选可用 |
| M8 | 搜索、通知、举报和管理后台可用 |
| M9 | 测试、安全和生产部署完成 |

第一轮开发的最重要停止点是 **M4**。达到 M4 后，映墨已经是一个真实可用的生活照片网站，再继续开发游戏教材区。

# 5. Codex 任务粒度

建议整个项目拆为约 30 个小任务，而不是 9 个大指令。

每个任务要求：

- 只解决一个明确目标；
- 修改范围可控；
- 有自动化测试；
- 有手动验收步骤；
- 不使用假数据冒充真实功能；
- 不留下未说明的 TODO；
- 完成后更新 `docs/progress.md`；
- 每次提交或 PR 都可独立回滚。

单个任务不应同时包含：

- 数据库架构大改；
- 后端多个模块；
- 多个复杂前端页面；
- 部署；
- 全站重构。

# 9. 项目开发原则

- 先日常区，后游戏区；
- 先闭环，后扩展；
- 先后端规则，后前端表现；
- 公共互动能力只开发一次；
- 重复检查以后端数据库约束为最终防线；
- 每阶段验收后再进入下一阶段；
- 旧静态版本用于参考，不长期双线维护；
- 不让 Codex 一次承担整个项目；
- 不接受固定返回值、空页面或硬编码数据作为完成结果。
