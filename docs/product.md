# 映墨 Yingmo 产品需求文档（PRD V2）

> 版本：V2.0  
> 产品形态：Web 内容社区  
> 前端：React + Vite  
> 后端：Flask + RESTful API  
> 核心空间：日常生活区、游戏教材区  
> 文档用途：作为产品、设计、开发、测试和验收的统一依据

---

## 1. 产品概述

### 1.1 产品定位

映墨是一个兼具生活影像记录与游戏知识共享的轻量内容社区。

用户可以：

1. 在日常生活区按照城市、景点、旅行或生活主题上传照片、记录日常；
2. 在游戏教材区按照游戏、英雄、地图和技巧类型，发布结构化点位教程；
3. 对内容点赞、评论、收藏；
4. 通过个人主页管理自己的内容和收藏；
5. 共同维护城市章节、英雄章节、地图章节和游戏知识。

### 1.2 产品核心价值

#### 日常生活区

- 用照片和文字保存旅行与普通生活；
- 按城市、景点和主题组织内容；
- 多个用户可以向同一章节持续投稿；
- 通过评论和互动形成共同回忆。

#### 游戏教材区

- 以“游戏—英雄—地图—教材”组织知识；
- 用户可以快速查找某个英雄在某张地图上的点位；
- 教材支持多张步骤图、站位说明、瞄准参照物和版本信息；
- 英雄和地图统一维护，避免重复章节；
- 用户可以反馈教材是否仍适用于当前版本。

## 2. 产品目标与范围

### 2.1 V2 第一阶段目标

完成真实可用的基本闭环：

```text
注册登录
→ 上传图片
→ 发布日常或游戏教材
→ 内容进入对应章节
→ 其他用户浏览、点赞、评论、收藏
→ 作者收到通知
→ 管理员处理章节与举报
```

### 2.2 V2 必须实现

- 用户注册、登录、退出和登录状态维护；
- 图片真实上传、缩略图和图片访问；
- 日常章节、日常发布、列表和详情；
- 游戏、英雄、地图和教材结构；
- 英雄与地图重复检查；
- 点赞、评论、收藏；
- 用户主页与个人中心；
- 基础通知；
- 内容举报；
- 基础管理后台；
- 响应式页面；
- 生产环境部署。

### 2.3 暂不实现

- 私信；
- 直播；
- 短视频编辑；
- 商业广告；
- 支付；
- 复杂关注与粉丝体系；
- 推荐算法；
- 原生移动端；
- 自动同步游戏战绩；
- AI 自动识别地图和英雄。

## 3. 用户角色

### 3.1 游客

可以浏览公开内容、搜索、查看用户公开主页和评论。

不能发布、点赞、评论、收藏、创建章节或举报。

### 3.2 注册用户

可以发布日常和游戏教材、上传图片、互动、收藏、申请新章节、编辑自己的内容、举报违规内容。

### 3.3 内容管理员

可以审核章节、合并重复章节、下架内容、删除违规评论、处理举报、标记教材失效。

### 3.4 系统管理员

在内容管理员权限基础上，可以管理用户、管理员、系统配置和操作日志。

## 5. 首页

首页负责产品介绍和双区域分流。

### 5.1 顶部导航

- 首页；
- 日常；
- 游戏教材；
- 发现；
- 发布；
- 通知；
- 个人头像；
- 深浅主题切换。

### 5.2 首屏双入口

#### 日常生活

文案方向：

> 记录旅行、照片和普通日子。

主要操作：

- 浏览日常；
- 发布今天。

#### 游戏教材

文案方向：

> 分享英雄技巧、地图点位和实战经验。

主要操作：

- 浏览教材；
- 发布点位。

- 最新日常；
- 热门城市或旅行章节；
- 最新游戏教材；
- 热门游戏、英雄或地图章节。

首页只做精选和分流，不承载完整内容管理。

## 6. 用户与认证

### 6.1 注册字段

- 用户名；
- 邮箱；
- 密码；
- 确认密码；
- 用户协议确认。

规则：

- 用户名 2～20 个字符；
- 用户名唯一；
- 邮箱唯一；
- 密码不少于 8 位；
- 密码必须哈希保存；
- 禁止保存明文密码。

### 6.2 登录

支持用户名或邮箱加密码登录。

后端采用 JWT，至少包含：

- Access Token；
- Refresh Token；
- Token 失效处理；
- 退出登录；
- 受保护接口鉴权。

### 6.3 用户资料

- 头像；
- 用户名；
- 昵称；
- 简介；
- 所在地区；
- 注册时间；
- 日常数量；
- 教材数量；
- 获赞数量。

### 6.4 用户状态

- 正常；
- 限制发布；
- 禁止评论；
- 封禁；
- 注销。

## 7. 日常生活区

### 7.1 日常章节

章节类型：

- 城市；
- 景点；
- 旅行；
- 校园；
- 活动；
- 自定义主题。

章节支持两级结构，例如：

```text
青岛
├── 五四广场
├── 栈桥
├── 八大关
└── 崂山
```

章节字段：

- 名称；
- 标准化名称；
- Slug；
- 类型；
- 上级章节；
- 国家、省份和城市；
- 简介；
- 封面；
- 创建人；
- 审核状态；
- 别名；
- 内容数；
- 参与用户数。

### 7.2 防止重复章节

用户创建章节前必须先搜索。

前端显示可能重复项，后端再次校验。标准化处理至少包括：

- 去除首尾空格；
- 全角半角统一；
- 大小写统一；
- 无意义字符清理；
- 别名匹配；
- 模糊匹配。

### 7.3 发布日常

必填：

- 至少一张图片；
- 标题；
- 所属章节。

选填：

- 正文；
- 地点；
- 心情；
- 标签；
- 拍摄时间；
- 可见范围。

图片规则：

- 每条最多 9 张；
- 支持 JPG、JPEG、PNG、WebP；
- 单张不超过 15MB；
- 第一张默认为封面；
- 支持排序；
- 后端必须校验真实文件类型；
- 自动生成缩略图。

可见范围：

- 公开；
- 仅登录用户；
- 仅自己。

### 7.4 日常详情

显示：

- 图片轮播；
- 标题和正文；
- 作者；
- 发布时间和拍摄时间；
- 所属章节；
- 地点、心情、标签；
- 点赞、评论、收藏、举报；
- 作者编辑和删除入口。

## 8. 游戏教材区

### 8.1 层级结构

```text
游戏
└── 英雄
    └── 地图
        └── 教材
```

不依赖英雄的地图通用教程可以直接归属游戏和地图。

### 8.2 游戏

游戏字段：

- 中文名；
- 英文名；
- 简称和别名；
- 图标和封面；
- 简介；
- 当前版本；
- 状态。

游戏正式数据原则上由管理员维护，普通用户可以提交新增申请。

### 8.3 英雄

字段：

- 所属游戏；
- 中文名；
- 英文名；
- 标准化名称；
- 别名；
- 头像；
- 英雄定位；
- 简介；
- 审核状态。

同一游戏中同一英雄只能存在一次：

```text
UNIQUE(game_id, normalized_name)
```

### 8.4 地图

字段：

- 所属游戏；
- 中文名；
- 英文名；
- 标准化名称；
- 别名；
- 地图类型；
- 封面；
- 简介；
- 当前状态；
- 审核状态。

同一游戏中同一地图只能存在一次：

```text
UNIQUE(game_id, normalized_name)
```

### 8.5 教材分类

- 技能点位；
- 炮台点位；
- 手雷投掷；
- 雷管投掷；
- 架枪位置；
- 防守站位；
- 进攻路线；
- 开局技巧；
- 能量获取；
- 阵容配合；
- 地图机制；
- 其他技巧。

### 8.6 发布游戏教材

必填：

- 游戏；
- 标题；
- 教材分类；
- 至少一张图片；
- 操作说明。

条件必填：

- 英雄专属教材必须选择英雄；
- 地图相关教材必须选择地图。

选填：

- 地图区域；
- 进攻方或防守方；
- 技能；
- 瞄准参照物；
- 使用时机；
- 难度；
- 游戏版本；
- 标签；
- 注意事项；
- 视频链接。

### 8.7 教材步骤

每个步骤包含：

- 顺序；
- 图片；
- 步骤标题；
- 步骤说明。

用户可以调整步骤顺序。

### 8.8 教材有效性

状态：

- 未验证；
- 当前有效；
- 可能失效；
- 已失效。

记录：

- 游戏版本；
- 测试日期；
- 有效性反馈；
- 最近确认时间。

### 8.9 教材筛选

- 游戏；
- 英雄；
- 地图；
- 分类；
- 阵营；
- 难度；
- 有效状态；
- 游戏版本。

## 9. 公共互动系统

### 9.1 点赞

- 支持日常和游戏教材；
- 同一用户对同一内容只能点赞一次；
- 再次点击取消；
- 产生通知。

### 9.2 收藏

- 支持日常和游戏教材；
- 在个人中心分类查看；
- 收藏人列表默认不公开。

### 9.3 评论

V2 首版支持：

- 一级评论；
- 一级回复；
- 评论删除；
- 分页加载。

评论长度 1～500 字。

### 9.4 举报

对象：

- 日常；
- 游戏教材；
- 评论；
- 用户。

原因：

- 违规低俗；
- 暴力违法；
- 骚扰辱骂；
- 广告垃圾；
- 盗用；
- 错误教学；
- 重复内容；
- 其他。

## 16. 后端 API 规划

统一前缀：

```text
/api/v1
```

主要接口组：

- `/auth`：注册、登录、刷新、退出、当前用户；
- `/users`：用户资料和个人内容；
- `/uploads`：图片上传和删除；
- `/life/chapters`：日常章节；
- `/life/posts`：日常内容；
- `/games`：游戏、英雄、地图；
- `/guides`：游戏教材和步骤；
- `/comments`：评论；
- `/interactions`：点赞和收藏；
- `/notifications`：通知；
- `/reports`：举报；
- `/admin`：后台管理。

接口必须统一：

- 状态码；
- 错误响应；
- 分页格式；
- 鉴权；
- 参数校验；
- 时间格式。

## 20. 核心验收标准

### 用户

- 可以注册、登录、刷新状态和退出；
- 重复用户名和邮箱被拒绝；
- 密码不以明文保存；
- 未登录用户不能发布。

### 日常

- 可以选择或申请章节；
- 可以上传真实图片并发布；
- 内容出现在对应章节；
- 作者可以编辑和删除；
- 私密内容不能被其他用户读取。

### 游戏教材

- 可以按游戏、英雄、地图浏览；
- 已有英雄和地图不能重复创建；
- 可以发布多步骤图文教材；
- 可以记录版本和有效状态；
- 可以按条件筛选。

### 互动

- 点赞不能重复；
- 评论和回复可用；
- 收藏可在个人中心查看；
- 互动产生通知。

### 管理

- 管理员可以审核章节、处理举报和下架内容；
- 普通用户无法访问管理接口；
- 章节合并后原内容正确迁移。

## 21. 产品原则

- 日常区强调影像、情绪和共同回忆；
- 游戏区强调结构、准确性和可学习性；
- 两个区域共享用户和互动能力，但不强行共用相同内容结构；
- 首页负责分流；
- 防重复规则以后端为最终依据；
- 优先完成真实闭环，不用静态假数据冒充完成；
- 所有功能必须可以测试和验收。
