# 映墨 V1 产品需求文档（PRD）

> 版本：V1.0  
> 日期：2026-07-02  
> 产品定位：给身边好朋友共同使用的生活照片记录与分享网站  
> 当前阶段：先改造静态 HTML，为后续照片画廊、上传、评论、用户系统打基础

---

## 1. 文档目的

本文档用于明确「映墨」第一版产品的需求边界、页面结构、核心功能和后续开发方向。

它不是商业竞争分析，也不是完整商业化方案。映墨的初衷不是和 Instagram、Google Photos、Flickr、Immich 等产品竞争，而是做一个小而温暖的网站，让身边的好朋友可以一起上传照片、记录生活、浏览彼此的日常，并留下轻量评论。

本文档后续可以交给 Codex / Cursor / Claude Code 等工具，用于指导前端页面改造与后续功能开发。

---

## 2. 产品背景

### 2.1 初衷

映墨希望承载的是一种很轻的生活记录：

- 朋友之间上传日常照片；
- 每张照片可以配一句话、一段心情或一个地点；
- 大家可以看到彼此生活中的小片段；
- 可以评论，但不追求强社交刺激；
- 可以浏览，但不做复杂推荐算法；
- 可以沉淀成个人时间线和共同回忆。

相比大型社交平台，映墨更像一个朋友之间的生活小院子。

### 2.2 调研启发

本次需求参考了几类成熟产品的常见模式：

| 产品类型 | 代表产品 | 对映墨的启发 |
|---|---|---|
| 照片社交 | Pixelfed / Instagram | 照片动态、相册、多图发布、评论、点赞、时间线 |
| 自托管相册 | Immich / PhotoPrism / Piwigo | 上传、照片管理、相册、标签、隐私权限、元数据 |
| 摄影社区 / 画廊 | Flickr / VSCO / Lychee / Photoview | 照片优先、网格展示、详情预览、分享体验 |

但映墨不直接复制这些产品。映墨第一版只取最适合小范围朋友使用的部分：照片墙、上传入口、生活动态、评论、个人主页、轻量权限。

## 3. 产品定位

### 3.1 一句话定位

**映墨是一个给朋友之间使用的温暖生活照片分享网站。**

用户可以上传一张或多张照片，写下一句日常，把普通生活片段整理成照片墙；也可以浏览朋友的生活瞬间，并留下轻量评论。

### 3.2 产品关键词

- 生活记录
- 照片分享
- 好友圈子
- 日常片段
- 温和互动
- 个人时间线
- 照片墙
- 低压力发布
- 轻量评论
- 共同回忆

### 3.3 非目标

第一版不做以下方向：

- 不做大型公开社交平台；
- 不做陌生人强推荐信息流；
- 不做短视频平台；
- 不做商业摄影作品集平台；
- 不做复杂 AI 相册管理；
- 不做无限功能的网盘或照片备份工具；
- 不做点赞排行榜、粉丝竞争、热榜等强刺激机制。

## 4. 目标用户

### 4.1 核心用户

第一阶段目标用户不是泛互联网用户，而是站长身边的好朋友。

典型用户包括：

1. **站长本人**  
   希望搭建并维护一个有个人风格的网站，记录生活，也让朋友参与进来。

2. **身边朋友**  
   愿意上传一些照片，看看朋友近况，偶尔评论互动。

3. **轻度浏览者**  
   不一定频繁发布，但会打开网站看看最近大家发了什么。

### 4.2 用户特点

- 不一定有强烈社交需求；
- 更在意真实、亲近、舒服的氛围；
- 不希望发布压力太大；
- 不需要复杂功能；
- 更接受“朋友小圈子”而不是完全公开平台。

## 5. 核心使用场景

### 场景 1：记录今天

用户看到一张照片，想把它传到映墨。

流程：

1. 打开网站；
2. 点击「上传照片」；
3. 选择图片；
4. 填写标题或一句话；
5. 可选填写地点、标签、心情；
6. 发布；
7. 照片出现在首页或照片墙。

### 场景 2：看看朋友最近在干什么

用户没有明确目的，只是想看看朋友们最近发了什么。

流程：

1. 打开首页；
2. 浏览最近照片动态；
3. 点击感兴趣的照片；
4. 查看大图、描述、评论；
5. 留下一句评论。

### 场景 3：回看某个人的生活时间线

用户想看看某个朋友最近记录了什么。

流程：

1. 点击照片卡片上的作者；
2. 进入用户主页；
3. 查看该用户上传的照片；
4. 按时间浏览生活片段。

### 场景 4：共同回忆

几位朋友在某次出游、聚餐、毕业、校园生活中产生了一组照片，希望统一放到一个页面里。

流程：

1. 创建相册或使用标签；
2. 多人上传相关照片；
3. 通过标签或相册集中浏览；
4. 评论补充回忆。

## 6. V1 产品范围

### 6.1 当前静态 HTML 阶段

当前阶段重点不是立刻完成真实后端，而是先把页面结构改对，让它从“课程/博客页面”变成“照片生活社区雏形”。

当前阶段应实现：

- 品牌导航；
- 首页 Hero；
- 上传入口；
- 照片墙；
- 照片动态卡片；
- 照片详情弹窗；
- 评论预览；
- 作者信息展示；
- 标签 / 心情 / 时间展示；
- 页脚；
- 本地静态图片数据结构升级。

### 6.2 后续动态功能阶段

后续逐步接入：

- 真实图片上传；
- 用户登录；
- 发布动态；
- 评论；
- 点赞；
- 用户主页；
- 相册；
- 可见范围控制。

## 8. 页面需求

## 8.1 首页

### 页面目标

首页负责传达映墨的气质，并让用户快速看到朋友们的生活照片。

### 核心模块

1. 顶部导航
2. Hero 区
3. 上传入口
4. 最近更新照片流
5. 照片墙
6. 评论 / 互动预览
7. 关于映墨
8. 页脚

### 建议导航

```text
首页 / 发现 / 上传 / 照片墙 / 关于映墨
```

不要再使用：

```text
课程 / 笔记 / 博客 / 实践 / Read More / Categories / Tags
```

### Hero 文案方向

可以使用类似表达：

```text
把日常映成墨色
上传一张照片，留下一句生活。
这里收藏朋友们普通日子里的光、风、饭香、路口和心情。
```

按钮：

- 主要按钮：上传今天的照片
- 次要按钮：看看朋友们的日常

---

## 8.2 照片墙

### 页面目标

照片墙是映墨的核心区域，用于展示用户上传的生活照片。

### 展示方式

第一版建议使用响应式网格：

- 桌面端：3 或 4 列；
- 平板端：2 或 3 列；
- 手机端：1 列或 2 列；
- 图片比例可以统一为 4:3，也可以后期升级为瀑布流。

### 照片卡片字段

每张照片卡片至少包含：

| 字段 | 是否必需 | 说明 |
|---|---|---|
| 图片 | 必需 | 照片主体 |
| 标题 | 必需 | 简短标题 |
| 描述 | 可选 | 一句话生活记录 |
| 作者 | 必需 | 发布者昵称 |
| 发布时间 | 必需 | 日期或相对时间 |
| 标签 | 可选 | 日常、城市、校园、食物等 |
| 点赞数 | 可选 | 后续互动 |
| 评论数 | 可选 | 后续互动 |

---

## 8.3 上传入口

### 页面目标

上传入口用于告诉用户：这个网站不是只能看，也可以参与记录。

### 静态阶段

当前没有后端时，可以先做一个上传入口卡片：

- 标题：记录今天
- 描述：选择一张照片，写下一句日常
- 按钮：上传照片
- 点击后可以先跳转到上传区域或弹出静态上传表单

### 后续动态阶段

上传表单字段建议：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| 图片 | file | 是 | 支持 jpg/png/webp |
| 标题 | text | 是 | 1-30 字 |
| 描述 | textarea | 否 | 0-200 字 |
| 标签 | text / multi-select | 否 | 可多选 |
| 地点 | text | 否 | 用户手动填写 |
| 可见范围 | select | 是 | 公开 / 好友可见 / 仅自己 |

---

## 8.4 照片详情弹窗

### 页面目标

点击照片后展示更完整的信息，不让用户离开当前页面也能看大图和评论。

### 内容结构

照片详情弹窗应包含：

- 大图；
- 标题；
- 描述；
- 作者；
- 时间；
- 地点；
- 标签；
- 点赞按钮；
- 评论列表；
- 评论输入框，后续实现。

### 交互要求

- 点击照片卡片打开弹窗；
- 点击遮罩或关闭按钮关闭；
- 按 Esc 关闭；
- 打开弹窗后页面背景不滚动；
- 移动端弹窗要适配屏幕高度。

---

## 8.5 评论区

### 页面目标

评论区用于朋友之间轻量互动，不追求复杂社交。

### 第一版展示

静态阶段可以展示 1-3 条评论预览。

字段包括：

| 字段 | 说明 |
|---|---|
| 评论者昵称 | 谁评论的 |
| 评论内容 | 一句话反馈 |
| 评论时间 | 可选 |

### 后续功能

- 登录后评论；
- 删除自己的评论；
- 评论字数限制；
- 简单敏感内容处理；
- 不做楼中楼，避免复杂化。

---

## 8.6 用户主页

### 页面目标

用户主页用于展示某个朋友自己的照片时间线。

### 第一版可以先不做独立页面

静态阶段可以只在照片卡片中展示作者信息，为后续用户主页预留链接。

### 后续结构

用户主页包含：

- 头像；
- 昵称；
- 简介；
- 照片数；
- 评论数或获赞数；
- 照片时间线；
- 相册集合。

## 12. 交互需求

### 12.1 照片卡片

- 鼠标悬停时轻微上浮；
- 显示“查看照片”提示；
- 点击卡片打开详情弹窗；
- 键盘 Enter / Space 也可打开；
- 图片加载失败时显示占位提示。

### 12.2 详情弹窗

- 打开后聚焦关闭按钮；
- Esc 关闭；
- 点击遮罩关闭；
- 关闭后焦点回到原照片卡片；
- 大图保持比例，不被裁切；
- 移动端内容可滚动。

### 12.3 上传入口

静态阶段：

- 点击后滚动到上传区域或打开静态上传卡片；
- 暂时不需要真实上传。

动态阶段：

- 选择图片后本地预览；
- 文件过大时提示；
- 发布成功后进入照片详情或返回照片墙；
- 发布失败时保留用户已填写内容。

## 13. 视觉与体验原则

映墨后续前端设计应遵循已有 UI 范式文档：

- 浅米色背景；
- 深绿色强调色；
- 霞鹜文楷屏幕阅读版字体；
- 宽松留白；
- 圆角卡片；
- 轻阴影；
- 温暖、克制、低饱和；
- 中文阅读友好；
- 小墨团 Logo 作为品牌识别。

特别注意：

- 页面要有生活气，不要像后台管理系统；
- 不要做成强商业社交平台；
- 不要做成摄影师作品集；
- 不要做成课程博客站；
- 不要堆砌复杂动画；
- 照片是主角，文字是辅助。

---

## 14. 权限与隐私原则

因为映墨是给朋友使用的网站，权限设计很重要。

### 14.1 第一阶段

静态阶段只需要在数据结构中预留：

```js
visibility: "public"
```

### 14.2 后续阶段

建议支持三种可见范围：

| 可见范围 | 说明 |
|---|---|
| public | 所有人可见 |
| friends | 登录后的好友可见 |
| private | 仅自己可见 |

### 14.3 后续需要注意

- 用户删除照片后，图片文件也应从服务器移除或标记不可访问；
- 未登录用户不能评论；
- 用户只能删除自己的照片和评论；
- 管理员可以删除不合适内容；
- 图片上传需要限制格式和大小。

## 15. 非功能需求

### 15.1 性能

- 首页首屏加载应尽量快；
- 图片应使用懒加载；
- 后续上传图片要压缩或生成缩略图；
- 照片墙不要一次性加载过多大图。

### 15.2 兼容性

- 支持主流桌面浏览器；
- 支持手机浏览器；
- 移动端上传入口要好用；
- 小屏幕弹窗不能溢出。

### 15.3 可维护性

- 静态数据结构尽量接近后端模型；
- class 命名要产品化，不要使用 course、post、article 等旧命名；
- CSS 变量统一管理颜色、字体、间距；
- 后续方便拆分成 Vue / React 组件。

### 15.4 安全

后续接入后端时需要：

- 限制上传文件类型；
- 限制上传文件大小；
- 防止恶意脚本注入；
- 评论内容需要转义；
- 登录态需要安全处理；
- 图片访问权限需要校验。

## 16. 成功标准

### 16.1 当前静态页面成功标准

- 首页不再像课程 / 博客 / 学习站；
- 用户一眼能看出这是生活照片分享网站；
- 照片墙能正常展示；
- 点击照片能打开详情；
- 页面中有明确上传入口；
- 页面中有评论和作者信息的结构雏形；
- 移动端可正常浏览；
- 视觉仍保持映墨的温暖文艺风格。

### 16.2 第一版可用产品成功标准

- 朋友可以注册或登录；
- 可以上传照片；
- 可以填写标题和描述；
- 照片会出现在照片墙；
- 其他朋友可以浏览和评论；
- 用户可以删除自己的照片；
- 网站在服务器上稳定运行。

## 20. 本版结论

映墨 V1 的重点不是做一个“大而全”的照片平台，而是先把一个朋友之间使用的小型生活照片网站做出来。

第一版的核心判断是：

> 一张照片 + 一句话 + 一个朋友身份 + 一点轻量评论，就足以构成映墨的最小产品闭环。

只要当前 HTML 先把这个闭环表达出来，后面再接上传、登录、评论和用户主页，就不会跑偏。
