# 映墨 Yingmo 产品需求文档（PRD V2.1）

> 版本：V2.1
> 产品形态：Web 内容社区  
> 前端：React + Vite  
> 后端：Flask + RESTful API  
> 核心空间：日常生活区、游戏点位区
> 文档用途：作为产品、设计、开发、测试和验收的统一依据
> 本次修订重点：在保持日常生活区需求不变的前提下，将游戏区明确为“地图优先、英雄次之、点位教程为核心”的快速检索与共享工具。

---

## 1. 产品概述

### 1.1 产品定位

映墨是一个兼具生活影像记录与游戏实用点位共享的轻量内容社区。

用户可以：

1. 在日常生活区按照城市、景点、旅行或生活主题上传照片、记录日常；
2. 在游戏点位区先选择当前游戏地图，再选择所用英雄，快速查找该英雄在该地图中的实用点位；
3. 上传由自己或朋友实际发现、验证和整理的点位教程，使用文字、图片和外部视频链接说明操作方法；
4. 对内容点赞、评论、收藏；
5. 通过个人主页管理自己的内容和收藏；
6. 共同维护城市章节和游戏点位知识。

### 1.2 产品核心价值

#### 日常生活区

- 用照片和文字保存旅行与普通生活；
- 按城市、景点和主题组织内容；
- 多个用户可以向同一章节持续投稿；
- 通过评论和互动形成共同回忆。

#### 游戏点位区

- 以“游戏—地图—英雄—点位教程”作为主要浏览路径；
- 服务于玩家已经进入某张地图、需要临时快速查找当前英雄点位的真实场景；
- 地图和英雄是平台统一维护的固定基础数据，用户不需要重复创建；
- 点位教程以简洁文字、多张有序图片和可选视频链接说明站位、方向、操作和效果；
- 普通点位保持较少必填项，特殊点位只增加与自身有关的少量字段；
- 开局定时投掷类点位必须记录投掷时间或投掷时机；
- 用户可以反馈点位是否仍适用于当前版本。

### 1.3 游戏点位区首期范围

V2 第一阶段的游戏点位区以《守望先锋》为首个正式游戏目录。

要求：

- 数据模型保留未来支持多个游戏的能力；
- 第一阶段前端主要围绕《守望先锋》的地图、英雄和点位教程设计；
- 当系统中只有一个启用中的游戏时，用户进入游戏区后可以直接进入该游戏的地图列表，不要求额外进行一次没有意义的游戏选择；
- 《守望先锋》的英雄和地图由管理员按照当前正式游戏数据统一录入、维护和更新；
- 普通用户不能自行创建正式英雄或正式地图；
- 第一阶段不建设英雄百科、赛事资讯、阵容模拟器、战绩查询或完整攻略文章系统。

## 2. 产品目标与范围

### 2.1 V2 第一阶段目标

完成真实可用的基本闭环：

```text
注册登录
→ 上传图片
→ 发布日常或游戏点位
→ 内容进入对应日常章节或“地图 + 英雄”点位列表

→ 管理员处理章节、游戏目录、点位有效性与举报
```

```text

→ 选择当前地图
→ 选择当前英雄
→ 查看该组合下的点位列表
→ 打开点位教程
→ 通过文字、图片或视频快速理解并复现
```

### 2.2 V2 必须实现

- 用户注册、登录、退出和登录状态维护；
- 图片真实上传、缩略图和图片访问；
- 日常章节、日常发布、列表和详情；
- 游戏、地图、英雄和点位教程结构；
- 管理员维护固定游戏、地图和英雄目录；
- 地图优先、英雄次之的点位浏览流程；
- 普通点位的低负担发布流程；
- 开局定时投掷类点位的投掷时间必填规则；
- 点位文字说明、多图说明、图片排序和外部视频链接；
- 点赞、评论、收藏；
- 用户主页与个人中心；
- 基础通知；
- 内容举报；
- 基础管理后台；
- 响应式页面；
- 生产环境部署。

### 2.3 暂不实现

- 私信；
- 直播；
- 站内视频上传和视频编辑；
- 商业广告；
- 支付；
- 复杂关注与粉丝体系；
- 推荐算法；
- 原生移动端；
- 自动同步游戏战绩；
- AI 自动识别地图和英雄；
- 英雄百科和技能数值百科；
- 阵容模拟、克制关系计算和比赛数据分析；
- 电竞赛事资讯；
- 长篇综合攻略文章系统；
- 由普通用户自由创建正式英雄或正式地图。

## 3. 用户角色

### 3.1 游客

可以浏览公开内容、搜索、查看用户公开主页和评论。

不能发布、点赞、评论、收藏、创建日常章节或举报。

游客可以按照地图和英雄浏览公开点位教程。

### 3.2 注册用户

可以发布日常和游戏点位教程、上传图片、添加外部视频链接、互动、收藏、申请新的日常章节、编辑自己的内容、反馈点位有效性、举报违规或错误内容。

注册用户不能直接创建、修改或删除正式游戏、地图和英雄基础数据。

### 3.3 内容管理员

可以审核日常章节、合并重复日常章节、维护游戏目录、维护地图和英雄、下架内容、删除违规评论、处理举报、标记点位失效、纠正点位关联关系和分类。

### 3.4 系统管理员

在内容管理员权限基础上，可以管理用户、管理员、系统配置和操作日志。

## 8. 游戏点位区

### 8.1 产品目标

游戏点位区用于帮助玩家在已经知道当前地图的情况下，快速找到自己所选英雄在该地图中的实用点位。

典型场景：

1. 玩家进入一局《守望先锋》，当前地图已经确定；
2. 玩家选择或准备选择某个英雄；
3. 玩家希望在较短时间内查看该英雄在当前地图中可直接使用的点位；
4. 玩家进入映墨游戏区，先选择地图，再选择英雄；
5. 系统展示该“地图 + 英雄”组合下的点位列表；
6. 玩家通过文字、图片或视频快速理解点位并在游戏中复现。

游戏点位区的重点不是完整讲解英雄机制，而是回答：

```text
我当前在哪张地图？
我准备使用哪个英雄？
这个英雄在这张地图有哪些点位？
点位在哪里？
应该朝向哪里？
应该怎样操作？
什么时候操作？
最终效果是什么？
这个点位现在是否仍然有效？
```

### 8.2 核心浏览层级

主要浏览路径：

```text
游戏
└── 地图
    └── 英雄
        └── 点位教程
```

示例：

```text
守望先锋
└── 国王大道
    ├── 托比昂
    │   ├── A 点屋檐隐藏炮台
    │   ├── 转角高台炮台
    │   └── 运载目标盲区炮台
    ├── 安娜
    │   ├── 防守方开局出门瓶
    │   ├── A 点高台瓶
    │   └── 转角团战瓶
    └── 其他英雄
```

数据模型中地图和英雄都是点位教程的关联对象，但前端主路径必须坚持“先地图、后英雄”。

不允许将主路径设计为：

```text
先选英雄
→ 再从英雄页寻找地图
```

英雄优先入口只能作为辅助入口，不能成为游戏区默认流程。

### 8.3 首期游戏范围

V2 第一阶段：

- 首个正式游戏为《守望先锋》；
- 系统底层保留多游戏能力；
- 游戏区首页在只有一个启用游戏时，直接展示《守望先锋》的地图；
- 后续新增其他游戏时，再展示游戏选择层；
- 不因未来多游戏能力增加当前用户的操作步骤。

### 8.4 固定基础数据原则

游戏、地图和英雄属于平台固定基础数据。

原则：

- 由管理员统一创建、导入、编辑、启用和停用；
- 普通用户不能直接创建正式游戏、地图或英雄；
- 普通用户发现基础数据缺失或错误时，可以提交反馈或举报，但不能直接写入正式目录；
- 初始数据以正式游戏当前可用内容为准；
- 游戏更新后，由管理员新增、修改或停用对应英雄和地图；
- 停用英雄或地图不能导致历史点位被物理删除；
- 历史点位仍需保留，但必须显示关联对象已停用或当前不可用；
- 同一游戏内英雄和地图必须进行标准化和唯一性校验。

### 8.5 游戏

游戏字段：

- 中文名；
- 英文名；
- 简称和别名；
- Slug；
- 图标和封面；
- 简介；
- 当前版本；
- 状态；
- 排序权重。

状态：

- 启用；
- 维护中；
- 停用。

同一游戏只能存在一个正式启用目录。

游戏正式数据仅由管理员维护。

### 8.6 地图

地图是游戏区第一层主要检索对象。

地图字段：

- 所属游戏；
- 中文名；
- 英文名；
- 标准化名称；
- Slug；
- 别名；
- 地图类型；
- 封面；
- 简介；
- 当前状态；
- 排序权重；
- 创建时间；
- 更新时间。

地图状态：

- 当前可用；
- 暂时轮换外；
- 已停用。

同一游戏中同一地图只能存在一次：

```text
UNIQUE(game_id, normalized_name)
```

地图列表要求：

- 默认优先展示当前可用地图；
- 支持按地图名称和别名搜索；
- 支持按地图类型筛选；
- 支持查看全部、当前可用和已停用地图；
- 地图卡片展示地图封面、名称、地图类型和点位数量；
- 地图卡片可以展示已有点位涉及的英雄数量；
- 点击地图进入地图详情和英雄选择页。

### 8.7 英雄

英雄是地图之后的第二层主要检索对象。

英雄字段：

- 所属游戏；
- 中文名；
- 英文名；
- 标准化名称；
- Slug；
- 别名；
- 头像；
- 英雄定位；
- 简介；
- 当前状态；
- 排序权重；
- 创建时间；
- 更新时间。

英雄状态：

- 当前可用；
- 暂时禁用；
- 已停用。

同一游戏中同一英雄只能存在一次：

```text
UNIQUE(game_id, normalized_name)
```

英雄选择要求：

- 用户进入某张地图后，系统展示该游戏的英雄列表；
- 已经在该地图拥有点位的英雄优先展示；
- 每个英雄显示该地图下的点位数量；
- 支持按英雄名称和别名搜索；
- 支持按英雄定位筛选；
- 没有点位的英雄仍可以展示，但应明确显示“暂无点位”；
- 选择英雄后进入该“地图 + 英雄”的点位列表；
- 页面必须持续显示当前地图上下文，避免用户忘记自己正在浏览哪张地图。

### 8.8 点位教程定义

点位教程是游戏区最小、最核心的内容单位。

每篇点位教程只描述一个相对明确、可复现的实用点位或操作。

示例：

- 托比昂在国王大道的一个隐藏炮台位置；
- 安娜在国王大道防守方的一个开局定时瓶；
- 黑百合在哈瓦那的一个架枪位置；
- 温斯顿在直布罗陀的一个跳跃路线；
- 秩序之光在某张地图中的一个传送门部署点；
- 狂鼠在某个入口的夹子或炸弹点位。

V2 点位教程必须关联：

- 一个游戏；
- 一张地图；
- 一个英雄。

不在 V2 点位教程中支持：

- 不关联地图的英雄通用长篇攻略；
- 不关联英雄的纯地图百科；
- 一篇教程同时关联多个地图；
- 一篇教程同时关联多个英雄；
- 将多个完全不同的点位塞入一篇大型综合文章。

复杂操作可以在同一个点位教程内使用多张图片或分步说明，但仍应围绕同一个点位目标。

### 8.9 点位分类

V2 第一阶段分类保持少量、明确、易判断。

正式分类：

1. 炮台与部署点位；
2. 技能投掷；
3. 开局定时投掷；
4. 架枪与站位；
5. 位移与路线；
6. 地图机制与交互；
7. 其他点位。

分类说明：

#### 炮台与部署点位

适用于：

- 托比昂炮台；
- 秩序之光传送面板或炮台；
- 其他需要部署实体的英雄技能。

#### 技能投掷

适用于：

- 不依赖严格开局时间的技能抛物线；
- 需要站位、朝向、参照物和落点说明的技能投掷。

#### 开局定时投掷

适用于：

- 根据回合开始、开门或敌方行进时间提前投掷的技能；
- 例如安娜防守方提前投掷生物手雷，使其在进攻方出门时落下。

选择该分类时，“投掷时间或时机”必须填写。

#### 架枪与站位

适用于：

- 远程英雄架枪位置；
- 防守站位；
- 高台、掩体和视野点位。

#### 位移与路线

适用于：

- 跳跃点；
- 位移技能路线；
- 绕后路径；
- 快速到达高台或特殊区域的路径。

#### 地图机制与交互

适用于：

- 与地图机关、可破坏物、跳板、移动平台或特殊空间有关的技巧。

#### 其他点位

用于无法归入以上分类但仍符合“地图 + 英雄 + 可复现点位”定义的内容。

管理员可以调整点位分类，但不能要求普通用户面对大量难以判断的细分类。

### 8.10 点位教程字段

基础必填字段：

- 地图；
- 英雄；
- 点位标题；
- 点位分类；
- 详细说明；
- 至少一种可视化说明。

至少一种可视化说明指：

- 至少一张图片；或
- 一个通过校验的外部视频链接。

推荐优先上传图片。

基础选填字段：

- 地图区域；
- 进攻方或防守方；
- 标签；
- 外部视频链接；
- 游戏版本；
- 测试日期；
- 注意事项。

不设置为全局必填的字段：

- 技能名称；
- 瞄准参照物；
- 难度；
- 起点；
- 终点；
- 落点；
- 英雄定位；
- 阵容；
- 对位关系；
- 终极技能组合；
- 游戏模式；
- 详细数值。

以上信息原则上可以直接写入“详细说明”，避免把发布页面拆成过多输入框。

### 8.11 类型专属字段

点位发布表单根据点位分类动态展示少量专属字段。

#### 炮台与部署点位

无额外必填字段。

可以选填：

- 部署位置说明；
- 敌方视角说明；
- 是否容易被发现；
- 适用攻防方。

这些内容优先允许写入详细说明，不强制拆分。

#### 技能投掷

无额外全局必填字段。

可以选填：

- 技能名称；
- 瞄准参照物；
- 落点说明；
- 使用时机。

#### 开局定时投掷

额外必填：

- 投掷时间或投掷时机。

“投掷时间或投掷时机”使用短文本字段，不限制为纯数字。

有效示例：

- 开门后约 2.5 秒；
- 倒计时归零后立即投掷；
- 听到出生点开门音效后投掷；
- 回合开始后约 4 秒；
- 敌方接近第一个转角前投掷。

可以选填：

- 技能名称；
- 瞄准参照物；
- 预计落点；
- 适用攻防方。

#### 架枪与站位

无额外必填字段。

可以选填：

- 主要视野方向；
- 掩体说明；
- 撤退路线。

#### 位移与路线

无额外必填字段。

可以选填：

- 起点；
- 终点；
- 操作时机。

#### 地图机制与交互

无额外必填字段。

可以选填：

- 触发条件；
- 地图机关名称。

类型专属字段未启用时，不得阻止用户发布。

### 8.12 详细说明文本区

详细说明是点位教程的主要文字输入区域。

要求：

- 必填；
- 支持普通换行；
- 支持基础安全格式；
- 不要求用户填写完整长文；
- 提示用户描述最关键的复现信息。

推荐占位提示：

```text
请描述：
1. 从哪里站位；
2. 面向哪里；
3. 如何放置、瞄准、投掷或移动；
4. 什么时候使用；
5. 最终会产生什么效果；
6. 有哪些容易失败的地方。
```

详细说明应允许用户用一段或几段自然文本完成全部描述，不要求将每条信息拆成独立表单字段。

### 8.13 图片与媒体

图片规则：

- 支持 JPG、JPEG、PNG、WebP；
- 单张不超过 15MB；
- 每篇点位最多 20 张图片；
- 支持拖拽或按钮排序；
- 后端校验真实文件类型；
- 自动生成缩略图；
- 未完成发布的未绑定图片需要清理；
- 图片必须属于当前发布用户；
- 第一张图片默认为列表封面；
- 用户可以为每张图片添加选填说明；
- 图片说明不要求填写标题。

推荐图片顺序：

```text
站位位置
→ 面向方向
→ 准星或参照物
→ 操作过程
→ 最终落点或实际效果
→ 敌方视角或隐藏效果
```

不强制所有教程都按照该顺序上传。

### 8.14 视频链接

V2 第一阶段不支持站内视频文件上传。

支持：

- 用户填写外部视频链接；
- 前端识别受支持的视频平台；
- 支持安全跳转；
- 条件允许时展示受控的视频预览；
- 不支持的平台仍可以显示为普通外部链接；
- 必须对 URL 协议、长度和格式进行校验；
- 禁止执行用户提供的脚本或任意嵌入代码。

视频链接为选填。

当点位没有图片但有合法视频链接时，可以发布；列表封面使用地图封面、英雄头像或系统生成的组合占位图。

### 8.15 简单模式与分步模式

发布页默认采用简单模式：

- 一个标题；
- 一个详细说明文本区；
- 多张有序图片；
- 每张图片可选说明；
- 一个可选视频链接。

复杂点位可以启用分步模式。

分步模式中每个步骤可以包含：

- 顺序；
- 图片；
- 步骤标题；
- 步骤说明。

用户可以调整步骤顺序。

规则：

- 分步模式不是发布点位的默认要求；
- 不强制每张图片都填写步骤标题；
- 不强制简单点位拆成多个步骤；
- 后端可以继续使用有序步骤或媒体关系保存内容，但前端不得因此增加不必要的填写负担。

### 8.16 发布点位流程

用户可以从以下入口发布：

- 游戏区统一“发布点位”入口；
- 某张地图的“为这张地图发布点位”入口；
- 某张地图某个英雄的“发布该英雄点位”入口。

上下文预填规则：

- 从游戏区统一入口进入时，需要选择地图和英雄；
- 从地图详情进入时，自动预填地图；
- 从“地图 + 英雄”点位列表进入时，自动预填地图和英雄；
- 已经确定的上下文不要求用户重复选择；
- 用户仍可以在提交前修改允许修改的字段。

基础发布流程：

```text
确认地图和英雄
→ 填写标题
→ 选择分类
→ 填写详细说明
→ 上传图片或填写视频链接
→ 根据分类填写必要的专属字段
→ 保存草稿或发布
```

发布校验：

- 地图和英雄必须属于同一个游戏；
- 地图和英雄必须存在；
- 地图和英雄不能处于被物理删除状态；
- 标题不能为空；
- 分类必须有效；
- 详细说明不能为空；
- 必须至少存在一张已上传图片或一个合法视频链接；
- 开局定时投掷必须填写投掷时间或时机；
- 图片不能重复绑定；
- 用户必须拥有所有待绑定图片；
- 受限用户不能发布。

### 8.17 点位列表

用户完成地图和英雄选择后进入点位列表。

页面顶部必须清晰展示：

- 当前游戏；
- 当前地图；
- 当前英雄；
- 返回地图英雄选择的入口；
- 发布当前组合点位的入口。

每张点位卡片至少展示：

- 封面图或组合占位图；
- 点位标题；
- 分类；
- 地图区域；
- 攻防方；
- 投掷时间或时机摘要，仅在存在时展示；
- 当前有效状态；
- 作者；
- 最近更新时间；
- 点赞数或收藏数。

点位列表必须支持：

- 关键词搜索；
- 分类筛选；
- 进攻方或防守方筛选；
- 地图区域筛选；
- 有效状态筛选；
- 最新发布排序；
- 最近更新排序；
- 热门排序；
- 分页。

第一阶段默认排序为“最近更新”。

页面不能再次要求用户选择游戏、地图和英雄。

### 8.18 点位详情

点位详情显示：

- 游戏；
- 地图；
- 英雄；
- 点位标题；
- 点位分类；
- 地图区域；
- 进攻方或防守方；
- 投掷时间或时机，仅在存在时展示；
- 详细说明；
- 有序图片；
- 单图说明；
- 外部视频链接或安全预览；
- 游戏版本；
- 测试日期；
- 有效状态；
- 最近确认时间；
- 标签；
- 作者；
- 发布时间和更新时间；
- 点赞、评论、收藏、有效性反馈和举报；
- 作者编辑和删除入口；
- 返回当前“地图 + 英雄”点位列表的入口。

阅读目标：

- 玩家进入详情后，应能在较短时间内理解站位、朝向、操作、时机和效果；
- 重要信息不能隐藏在多层折叠组件中；
- 图片查看支持放大；
- 移动端图片和文字需要保持清晰；
- 投掷时间等关键字段应放在详情顶部可快速看到的位置。

### 8.19 地图详情与英雄选择

地图详情是游戏区最重要的中间页面。

地图详情显示：

- 地图名称；
- 地图封面；
- 地图类型；
- 地图简介；
- 当前状态；
- 该地图的点位总数；
- 该地图已有点位的英雄数量；
- 英雄选择区域；
- 最新点位或精选点位。

英雄选择区域要求：

- 头像优先；
- 中文名清晰可见；
- 显示该地图下的点位数量；
- 有点位的英雄优先排序；
- 支持英雄名称搜索；
- 支持英雄定位筛选；
- 支持“只看已有点位英雄”；
- 点击英雄进入当前地图下该英雄的点位列表；
- 不能先跳转到脱离当前地图上下文的通用英雄详情页。

### 8.20 辅助英雄页

系统可以保留英雄资料页作为辅助入口。

英雄页可以展示：

- 英雄头像；
- 名称；
- 定位；
- 简介；
- 拥有点位的地图列表；
- 每张地图的点位数量。

从英雄页点击地图后，必须进入：

```text
该地图 + 该英雄的点位列表
```

英雄页不是游戏区主入口，不得破坏地图优先原则。

### 8.21 点位有效性

状态：

- 未验证；
- 当前有效；
- 可能失效；
- 已失效。

记录：

- 游戏版本；
- 测试日期；
- 有效性反馈；
- 最近确认时间；
- 状态修改人；
- 状态修改时间。

规则：

- 新发布点位默认状态为“未验证”；
- 作者可以填写游戏版本和测试日期；
- 注册用户可以反馈“仍然有效”或“可能失效”；
- 同一用户不能对同一点位重复提交同类型有效性反馈；
- 内容管理员可以标记当前有效、可能失效或已失效；
- 地图、技能或游戏版本发生变化时，管理员可以批量将相关点位标记为可能失效；
- 已失效点位默认仍可访问，但必须有明显提示；
- 已失效点位不应出现在默认优先结果顶部；
- 被认定为错误、违规或危险利用的内容可以单独下架，不等同于“已失效”。

### 8.22 点位筛选

核心筛选：

- 地图；
- 英雄；
- 分类。

在地图和英雄已经由页面上下文确定后，不重复显示对应选择器。

辅助筛选：

- 地图区域；
- 进攻方或防守方；
- 有效状态；
- 游戏版本；
- 标签。

第一阶段不将“难度”作为核心筛选条件。

原因：

- 难度具有较强主观性；
- 玩家临时查点位时更关心地图、英雄、类型和攻防方；
- 减少发布字段和筛选负担。

### 8.23 搜索与快速检索

游戏区搜索支持：

- 地图中文名；
- 地图英文名；
- 地图别名；
- 英雄中文名；
- 英雄英文名；
- 英雄别名；
- 点位标题；
- 点位说明；
- 标签；
- 地图区域。

搜索结果按以下类型分组：

- 地图；
- 英雄；
- 点位教程。

点位搜索结果必须同时展示地图和英雄，避免用户只看到标题却无法判断适用场景。

快速检索目标：

- 用户从游戏区首页进入点位列表，核心操作只需要完成地图和英雄两次选择；
- 地图和英雄选择在移动端也必须容易点击；
- 不使用多层下拉菜单隐藏主要路径；
- 不要求用户先填写复杂筛选条件；
- 页面应尽量保留最近使用的地图和英雄上下文，但不得依赖本地状态才能正常使用；
- URL 必须包含可分享和可恢复的地图、英雄或点位上下文。

### 8.24 草稿

点位教程支持草稿。

草稿保存：

- 地图；
- 英雄；
- 标题；
- 分类；
- 详细说明；
- 类型专属字段；
- 图片顺序；
- 图片说明；
- 视频链接；
- 标签；
- 其他选填项。

规则：

- 草稿只对作者可见；
- 草稿不要求满足发布校验；
- 从草稿发布时再次执行完整校验；
- 删除草稿时清理不再被其他内容使用的未绑定媒体；
- 自动保存属于 P1，手动保存草稿属于 P0。

### 8.25 空状态、加载和错误状态

必须提供：

- 地图列表加载状态；
- 英雄列表加载状态；
- 点位列表加载状态；
- 点位详情加载状态；
- 无搜索结果状态；
- 当前地图暂无点位状态；
- 当前“地图 + 英雄”暂无点位状态；
- 图片加载失败状态；
- 视频链接不可用状态；
- 权限不足状态；
- 内容已下架状态；
- 关联地图或英雄已停用状态；
- 网络错误和重试入口。

空状态文案应直接引导用户贡献，例如：

> 这个英雄在这张地图还没有点位，记录第一个实用位置吧。

### 8.26 移动端要求

游戏点位区需要重点适配玩家在游戏过程中使用手机临时查询的场景。

要求：

- 地图卡片和英雄头像具有足够点击区域；
- 地图和英雄选择不依赖鼠标悬浮；
- 当前地图和英雄上下文在列表页清晰可见；
- 点位关键字段优先展示；
- 详情图片支持触摸查看；
- 不使用过宽表格；
- 筛选面板可以收起；
- 发布页面采用逐区块布局；
- 图片上传、排序和删除在触屏下可操作；
- 页面首屏避免被大面积装饰性 Hero 占据；
- 从地图进入英雄、再进入点位的路径必须清晰且容易返回。

### 8.27 游戏区不做什么

V2 游戏点位区不建设：

- 完整英雄百科；
- 技能伤害和冷却数值数据库；
- 英雄克制排行榜；
- 阵容搭配模拟器；
- 战绩查询；
- 实时对局数据；
- 电竞资讯；
- 赛事直播；
- 长篇综合攻略；
- 通用论坛；
- 与地图和英雄无关的泛游戏动态；

## 16. 后端 API 规划

统一前缀：

```text
/api/v1
```

主要接口组：

- `/auth`：注册、登录、刷新、退出、当前用户；
- `/users`：用户资料和个人内容；
- `/uploads`：图片上传和删除；
- `/life/chapters`：日常章节；
- `/life/posts`：日常内容；
- `/games`：游戏、英雄、地图；
- `/guides`：游戏点位教程、媒体和步骤；
- `/comments`：评论；
- `/interactions`：点赞和收藏；
- `/notifications`：通知；
- `/reports`：举报；
- `/admin`：后台管理。

游戏目录读取接口至少支持：

```text
GET /games
GET /games/:gameSlug
GET /games/:gameSlug/maps
GET /games/:gameSlug/maps/:mapSlug
GET /games/:gameSlug/heroes
GET /games/:gameSlug/heroes/:heroSlug
```

点位列表接口至少支持：

```text
GET /guides?game_id=&map_id=&hero_id=
```

其中：

- `game_id`、`map_id`、`hero_id` 是地图 + 英雄点位列表的核心参数；
- 分类、攻防方、区域、有效状态、版本、关键词和排序作为辅助参数；
- 后端必须校验地图和英雄属于同一游戏；
- 前端不能只依赖本地过滤完成权限和关联校验。

点位创建和更新接口必须校验：

- 当前用户权限；
- 游戏、地图和英雄存在；
- 地图和英雄属于同一游戏；
- 标题和说明；
- 媒体或视频链接；
- 分类；
- 开局定时投掷的 timing；
- 图片所有权和绑定状态；
- 草稿转换；
- 删除和更新时的媒体清理。

接口必须统一：

- 状态码；
- 错误响应；
- 分页格式；
- 鉴权；
- 参数校验；
- 时间格式。

## 20. 核心验收标准

### 用户

- 可以注册、登录、刷新状态和退出；
- 重复用户名和邮箱被拒绝；
- 密码不以明文保存；
- 未登录用户不能发布。

### 日常

- 可以选择或申请章节；
- 可以上传真实图片并发布；
- 内容出现在对应章节；
- 作者可以编辑和删除；
- 私密内容不能被其他用户读取。

### 游戏基础数据

- 系统中存在可用的《守望先锋》游戏目录；
- 管理员可以维护地图和英雄；
- 普通用户不能创建正式地图和英雄；
- 同一游戏内重复地图和英雄被后端拒绝；
- 停用地图或英雄后历史点位仍然保留；
- 地图和英雄别名可以参与搜索；
- 地图和英雄状态可以正确展示。

### 地图优先浏览

- 用户进入游戏区后可以优先看到地图列表；
- 当只有一个启用游戏时，不强制用户进行多余的游戏选择；
- 用户选择地图后可以看到英雄列表；
- 英雄卡片显示当前地图下的点位数量；
- 有点位的英雄优先展示；
- 用户选择英雄后进入该地图和英雄组合的点位列表；
- URL 可以恢复和分享地图、英雄上下文；
- 不需要通过通用英雄详情页才能进入地图点位。

### 点位发布

- 用户可以从地图 + 英雄上下文直接发布点位；
- 地图和英雄自动预填时不要求重复选择；
- 点位必须关联同一游戏下的一张地图和一个英雄；
- 普通点位只需要少量基础必填项；
- 用户可以通过一个详细说明文本区描述站位、方向、操作、时机和效果；
- 用户可以上传多张图片并排序；
- 单张图片说明可以留空；
- 用户可以添加外部视频链接；
- 至少一张图片或一个合法视频链接存在时才可发布；
- 开局定时投掷未填写投掷时间或时机时不能发布；
- 其他分类不要求填写投掷时间；
- 未绑定或无权限图片不能发布；
- 作者可以编辑和删除自己的点位；
- 草稿可以保存未完成内容。

### 点位查看与检索

- 用户可以按地图、英雄和分类查找点位；
- 进入组合点位列表后不重复要求选择地图和英雄；
- 点位卡片清晰展示地图、英雄、标题、分类和有效状态；
- 点位详情展示文字、图片和可选视频；
- 投掷时间在存在时清晰展示；
- 图片支持放大查看；
- 外部视频不可用时不影响文字和图片；
- 无点位时显示明确空状态和发布入口；
- 搜索结果中的点位同时显示地图和英雄；
- 移动端可以完成地图选择、英雄选择、点位查看和图片浏览。

### 点位有效性

- 新点位默认显示未验证；
- 管理员可以标记当前有效、可能失效和已失效；
- 已失效点位具有明显提示；
- 用户有效性反馈不能重复提交；
- 游戏版本和测试日期可以记录；
- 地图或英雄停用时相关点位显示关联状态；
- 点位失效与内容下架是两个不同状态。

### 互动

- 点赞不能重复；
- 评论和回复可用；
- 收藏可在个人中心查看；
- 点位收藏展示地图和英雄；
- 互动产生通知。

### 管理

- 管理员可以审核日常章节、处理举报和下架内容；
- 管理员可以维护游戏、地图和英雄；
- 管理员可以修正点位地图、英雄和分类；
- 管理员可以标记点位失效；
- 普通用户无法访问管理接口；
- 日常章节合并后原内容正确迁移；
- 停用游戏目录数据不会误删历史点位；
- 高风险管理操作写入操作日志。

## 21. 产品原则

- 日常区强调影像、情绪和共同回忆；
- 游戏区强调快速检索、准确关联和可直接复现；
- 游戏区主路径始终是“地图优先、英雄次之、点位为核心”；
- 地图和英雄是固定基础目录，点位教程是用户持续积累的核心内容资产；
- 游戏区优先解决玩家在对局中临时查点位的问题，而不是建设完整游戏百科；
- 普通点位保持少量必填项；
- 能用一段自然文本表达的信息，不强制拆成大量字段；
- 特殊分类只增加真正必要的专属字段；
- 开局定时投掷必须记录投掷时间或投掷时机；
- 图片和视频用于帮助玩家快速复现，而不是装饰内容；
- 两个区域共享用户和互动能力，但不强行共用相同内容结构；
- 首页负责分流；
- 防重复和关联一致性以后端为最终依据；
- 优先完成真实闭环，不用静态假数据冒充完成；
- 所有功能必须可以测试和验收。
